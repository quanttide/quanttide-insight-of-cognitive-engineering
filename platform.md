# 认知工程实践的核心洞察

从平台（qtcloud-think）与实验室（project-11）的实际构建中提取，非设计文档，而是做中学到的。

## 1. 格式即接口

报告模板和 gallery YAML 的 schema 不是文档问题——数据生产者（gallery）的格式直接决定了消费者（report generator）的实现。每改一次格式就要改一次代码。format 是 data flow 的契约，不是 UI 问题。

## 2. LLM 是增强，不是核心

系统在无 API 时也必须产出完整的可用输出。schematic usage、causals、boundaries、intention hierarchy 这些结构化数据本身包含足够信号用于有意义的关系计算和摘要生成。把基础路径做扎实，比依赖 LLM 填充占位符更重要。

## 3. 正交维度自然涌现

平台（动词管线：Clarify→Organize→Distill→Express）和实验室（名词分层：情境→意图→图式）是独立构建的，但最终呈现为同一二维空间的两个轴。没有人设计过这个结构——它是从实践中浮现出来的。说明认知工程的方法论（怎么加工）和数据模型（加工成什么）天然正交。

## 4. 先清理，后建设

每次有意义的添加之前都伴随删除：旧模板删除后才写新模板、错误的 AGENTS.md 删除后才重写、心智模型误判纠正后才继续。这不是预先规划的流程，而是被执行的模式——不接受在混乱上叠加新东西。

## 5. 元层不可绕过

两个系统各自独立得出同一结论：认知工程工具必须能反思自己的产出。平台的 Meta agent、实验室的"元"领域 + tension/coverage 命令，都不是设计文档要求的——是在使用中发现必须有的。

## 6. 数据模型编码关系信息

利用意图层级（top→bottom）计算领域间的张力关系，没有用到任何语义理解，纯粹从数据结构推导。说明数据模型的设计质量直接决定了可计算的分析深度——好的模型本身就有分析能力。

## 7. 认知产品需要审核状态

平台的 received/pending/rejected 和实验室的意图确认机制指向同一个需求：不是所有加工过的数据都能被接受为事实。"暂不接受"是认知工程的合法状态。

## 8. Distill 断层是方法问题，不是技术问题

从平台的笔记输出到实验室的 gallery YAML 之间的自动化映射缺失，不是因为实现难度高，而是因为精炼方法论本身未定义。Clarify 之后的"结构化"具体怎么做——agenda/ecology/frame 从笔记的哪个字段来？Intention 的 agent/level/trigger 怎么确定？这需要方法论先行。

## 9. 领域顺序是设计决策

contract/domain.yaml 定义的领域排序不是实现细节，它承载语义——什么领域优先、什么领域在后，影响报告的可读性和分析者的认知负担。把这个信息从代码外部化到配置文件中是正确做法。

## 10. 子模块工作流有真实成本

每次提交 = 子模块提交 + 主仓库提交 = 两倍操作。这个摩擦不是理论上的——每一次 git push 都要做两次。如果频繁修改子模块内容，这个成本会累积到影响工作流的流畅度。
